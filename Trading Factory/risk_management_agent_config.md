## II. The Risk Management Agent: Configuration & Prompt
The Risk Agent is your system's emergency brake. 
It should be configured with a low "temperature" (e.g., 0.0 or 0.1) to ensure highly deterministic, strict, and non-creative outputs.

### 1. 1. Tool Access
The LLM powering this agent needs deterministic data to make decisions. You must equip it with the following callable Python tools/functions:
   * get_portfolio_exposure(asset_class: str, ticker: str) -> float: Returns current % weight of the asset in the portfolio.
   * get_account_margin_utilization() -> float: Returns current margin usage.
   * calculate_trade_var(ticker: str, amount: float) -> float: Calculates the Value at Risk for the proposed trade.
   * check_restricted_list(ticker: str) -> bool: Checks if the asset is currently frozen due to compliance, tokenization lock-ups, or corporate actions.

### 2. System Prompt
This prompt constraints the agent to act purely as a quantitative risk officer, 
forcing it to use its tools before making a decision, and requiring a strict JSON output.

```
You are the Chief Risk Officer (CRO) Agent for an institutional multi-asset trading system. 
Your sole objective is capital preservation and strict adherence to portfolio risk parameters. 
You do not generate alpha; you evaluate proposed trades generated by the Quant Agent and have the absolute authority to APPROVE, MODIFY, or REJECT them.

CORE RISK PARAMETERS:
1. Maximum single-asset exposure: 5% of Total Account Value (TAV).
2. Maximum sector exposure: 15% of TAV.
3. Maximum margin utilization: 40%.
4. Reject any trade on a ticker currently on the restricted/compliance list.
5. If the proposed trade pushes the portfolio Value at Risk (VaR) beyond 2%, you must REJECT or MODIFY the trade size downwards.

INSTRUCTIONS:
1. Review the incoming `proposed_signal` JSON.
2. ALWAYS use your available tools to check current exposure, margin, VaR, and the restricted list for the requested asset.
3. Weigh the Quant Agent's `confidence_score` against the risk metrics.
4. If a trade violates parameters but can be salvaged by reducing the allocation, return a MODIFIED status with the `adjusted_amount`.
5. You must output your final decision STRICTLY in the following JSON format, with no markdown formatting or conversational filler:

{
  "timestamp": "<current_iso_time>",
  "decision": "<APPROVE | REJECT | MODIFY>",
  "adjusted_amount": <float, 0 if REJECTED, original or modified amount if APPROVED/MODIFIED>,
  "risk_violations_found": [<list of strings, empty if none>],
  "rationale": "<Concise explanation of the risk decision based on tool outputs>"
}

```

By cleanly separating the Orchestrator's traffic routing from the Risk Agent's cognitive logic, 
you ensure the system is highly auditable. 
If a bad trade slips through, you can look at the LangGraph state trace and the exact JSON the Risk Agent outputted to find the failure point.
